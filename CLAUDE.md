# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

전우찾기 / ComradeFind — a Korean-market landing site for ComradeFind's military life guide service. Features include a discharge D-Day calculator, discharge-prep checklist, comrade-matching signup, expert consultation booking, and an internal admin console.

## Commands

The repo ships both `bun.lockb` and `package-lock.json`; treat **bun** as primary and only fall back to npm if bun is unavailable.

```bash
bun install            # install deps
bun dev                # vite dev server on http://localhost:8080 (host "::")
bun run build          # production build
bun run build:dev      # dev-mode build
bun run preview        # serve dist/
bun lint               # eslint .
bun test               # vitest run (one-shot)
bun run test:watch     # vitest watch
```

Run a single vitest file or test name:

```bash
bunx vitest run src/path/to/file.test.ts
bunx vitest run -t "describes a behavior"
```

Playwright config (`playwright.config.ts`) is set up for local browser checks. There is no local `e2e/` suite checked in yet.

## Environment

`src/integrations/supabase/client.ts` reads three Vite env vars and **must not be edited by hand** (the file is regenerated by Supabase tooling):

- `VITE_SUPABASE_URL`
- `VITE_SUPABASE_PUBLISHABLE_KEY`
- `VITE_SUPABASE_PROJECT_ID`

The repo currently ships a committed `.env` for local dev. The Supabase project id is also pinned in `supabase/config.toml`.

## Architecture

### Stack

Vite 5 + React 18 + TypeScript (SWC via `@vitejs/plugin-react-swc`) · TailwindCSS + shadcn/ui (slate base, see `components.json`) · TanStack Query · React Router v6 · Supabase (auth + Postgres + edge functions + realtime) · framer-motion · react-hook-form + zod · vitest + Testing Library (jsdom).

Path alias `@/*` → `src/*` (configured in `vite.config.ts`, `vitest.config.ts`, and `tsconfig.json`). TypeScript is intentionally loose: `strictNullChecks: false`, `noImplicitAny: false`, `noUnusedLocals/Parameters: false`, `@typescript-eslint/no-unused-vars: off`. Don't tighten these globally without coordination — large parts of the codebase rely on the relaxed mode.

Build splits manual chunks for `react-vendor`, `supabase`, `ui-motion` (framer-motion), and `tanstack` (`vite.config.ts`).

### App shell and route gating (`src/App.tsx`)

`App` wires providers in this order: `QueryClientProvider` → `LanguageProvider` → `TooltipProvider` → `BrowserRouter`. The `QueryClient` defaults are tuned to avoid refetch storms (`staleTime: 5 min`, `gcTime: 30 min`, `refetchOnWindowFocus: false`, `retry: 1`) — preserve these defaults when adding queries unless the data is genuinely live.

`AppRoutes` reads `site_settings.service_enabled` (row id `"main"`) on mount **and** subscribes to a realtime `postgres_changes` UPDATE channel on the same row. When the flag is false, every route except `/admin` falls through to `Maintenance`. This is a kill-switch the admin console can flip from production — keep `/admin` reachable in any new gating logic.

### Code-splitting strategy

Only `Index` is eagerly imported. Every other page (`Admin`, `Maintenance`, `NotFound`, `Legal`, `FAQ`, `About`) is `React.lazy`. `Index.tsx` further splits below-the-fold sections (`BuddySearch`, `ExpertMatchSection`, `ServiceOverview`, `Footer`) and all modals (`FreeConsultModal`, `ServiceInfoModal`, `WelcomeModal`) so the initial render only ships `Navbar` + `DdayHeroSection` + `DischargeChecklist`. New heavy components should follow the same `lazy()` + `Suspense fallback` pattern.

### i18n (`src/contexts/LanguageContext.tsx`, `src/i18n/translations.ts`)

Four locales — `ko` (default), `en`, `ja`, `zh`. Translations live in a single dictionary keyed by dot.notation strings (e.g. `nav.consult`, `hero.title1`). `t(key, params?)` interpolates `{name}` placeholders and falls back to the Korean string, then to the raw key. `lang` persists in `localStorage` under `lv-lang`, and `setLang` deliberately calls `window.location.reload()`. The trailing `import.meta.hot.invalidate()` is also intentional: HMR otherwise duplicates the Context module and breaks `useLanguage` consumers. Don't replace either with partial updates.

`<html lang>` and `document.title` are synced from a per-language `TITLE_BY_LANG` map.

### Supabase integration

- Client (`src/integrations/supabase/client.ts`): `persistSession: true`, `autoRefreshToken: true`, `storage: localStorage`. Auto-generated, do not edit.
- Types (`src/integrations/supabase/types.ts`): generated `Database` type — also do not hand-edit. Regenerate via Supabase CLI when schema changes.
- Migrations: `supabase/migrations/*.sql` — append timestamped files; never rewrite history. Core tables: `consultations`, `experts`, `site_settings`, `admin_users`, `notifications`, `buddy_waiting_users`, `buddy_matches`.
- Edge functions: `supabase/functions/{create-admin,setup-admin,submit-consultation}/index.ts`. The first two bootstrap admin accounts; `submit-consultation` is the production form handler.

### shadcn/ui

`src/components/ui/` is generated shadcn output (slate base, no prefix). Use `bunx shadcn@latest add <component>` with the existing `components.json` rather than hand-rolling primitives. Aliases per `components.json`: `@/components`, `@/components/ui`, `@/lib/utils`, `@/lib`, `@/hooks`.

### Testing

`vitest` with jsdom environment, globals on, setup at `src/test/setup.ts`. Test glob: `src/**/*.{test,spec}.{ts,tsx}`. Co-locate tests next to the unit under test or place under `src/test/`.

## Conventions worth preserving

- **Don't introduce a global state library** — TanStack Query handles server state and React local state covers the rest.
- **Don't replace the maintenance gate's realtime subscription with polling.** Admins expect the toggle to take effect on every open tab within seconds.
- **Modals are lazy and gated by `open && <Suspense>...`** — the conditional wrapper is what keeps the chunk out of the initial bundle. Mounting them unconditionally re-introduces the bundle bloat the splitting was added to fix.
- **Use `build:dev` for development-mode production builds** when debugging mode-specific behavior.
